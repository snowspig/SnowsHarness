# 代码审查清单

## 严重度分级

每个发现必须按以下分级，且包含：文件路径、行号、问题、为何重要、修复建议。无证据不提发现。

- **Blocker**：会导致生产故障（安全漏洞、数据丢失、崩溃）
- **High**：很快会出问题（竞态、缺失错误处理）
- **Medium**：应修但不紧急（代码坏味道、缺测试、命名）
- **Nitpick**：风格偏好、轻微可读性改进

## 正确性

- [ ] 逻辑错误或 off-by-one
- [ ] 边界情况未处理（空输入、null、边界值）
- [ ] async/并发代码的竞态条件
- [ ] 资源泄漏（未关闭的文件、连接、句柄）
- [ ] 错误处理：是否捕获、记录、正确传播？

## 安全

- [ ] 输入校验与消毒
- [ ] SQL 注入 / 命令注入 / XSS
- [ ] 硬编码密钥或凭据
- [ ] 认证/授权检查缺失
- [ ] 不安全的反序列化

## 性能

- [ ] 不必要的 O(n²) 或更差算法
- [ ] N+1 查询模式
- [ ] 查询字段缺索引
- [ ] 无界内存使用（整文件/列表加载）
- [ ] async 上下文中的阻塞 I/O

## 代码质量

- [ ] 函数超过 50 行
- [ ] 嵌套超过 3 层
- [ ] 无命名常量的魔法数字
- [ ] 重复代码（DRY 违规）
- [ ] 缺类型注解（Python）/ 类型标注（TypeScript）

## 风格

- [ ] 是否遵循 PEP 8 / 项目 lint 规则
- [ ] 命名约定是否一致
- [ ] 变量函数名是否有意义

## 输出格式

```
[SEVERITY] file:line — <摘要>
  <解释>
  建议: <如何修复>
```

## 审查原则

- 引用精确的文件路径和行号
- 解释 WHY 是问题，而不只是 WHAT
- 给出具体的修复建议，而非模糊建议
- 承认发现的好模式
- linter 已覆盖的风格问题不要 nitpick
